## 1. 技能定位


| 项          | 内容                                                                                     |
| ---------- | -------------------------------------------------------------------------------------- |
| **名称**     | `using-agentsmd`                                                                       |
| **目的**     | 让 Agent 在**创建或维护** `AGENTS.md` 时，理解其作用、边界与编写方式                                         |
| **非目标**    | 不替代 `skill-creator` / `create-rule`；不提供冗长模板或围栏 checklist；不单独规范安全章节；不讨论运行时注入/再 read 等行为 |
| **风格**     | 轻量、判断导向；教心智模型 + 少量正反例，不写成第二份冗长规范                                                       |
| **技能正文语言** | 中文首版（`skills/using-agentsmd/SKILL.md`）；英文版待定 |


### 触发策略

**平时不加载**

- Agent 日常干活时，工具已被动注入 `AGENTS.md`；**照文件内容执行即可，无需加载本技能**

**必须加载（Hard gate）**

仅在 **创建或修改** `AGENTS.md`（根或嵌套）时：

- **必须先加载并遵循本技能**，再动手写文件
- 用户明确要求写/改/审计/`/init` 生成该文件 → 加载
- 用户说「把这条写进 AGENTS.md」→ 加载；仍先做归属判断，用户坚持后再照做
- Agent 自己打算 `create` / `edit` / `append` 该文件（含先征得许可后）→ 加载

**不触发**

- 仅阅读、遵循、引用 `AGENTS.md` 中的约定去干别的任务
- 仓库配置变更时**口头提醒**「可能要同步 AGENTS.md」——提醒不必加载；若随后要改文件，再加载

## 2. 核心定义（定调）

`AGENTS.md` 是：

- **以仓库为单位的、被动加载的高信号 prompt 片段**
- 不是给人看的说明文档，而是**考验提示词功底**的常驻上下文
- 编写得当可：**少犯错、走快路（高效）、减少人类重复沟通**

`AGENTS.md` 不是：

- 项目百科全书、变更日志、流水记账本
- Skill / Rule / README / docs 的复制品
- 通用 coding 准则或 user rules 的堆砌

### 收束表述

> 仓库级、被动加载的高信号 **prompt**：用提示词技巧编写。保留不知道就容易做错的红线、不知道就会做慢/做偏的绿线。持续删减；根精炼、差异嵌套；不写理由、不记账、不复制。体积靠写作规范与拆分约束，不靠行数 KPI——真写长了说明早该嵌套拆分。

## 3. 类别错误：文档 vs 提示词

Agent 写 `AGENTS.md` 时常误以为「这是给人看的文档」，于是写解释、对比、背景叙事。

**正确心态：按写 prompt 的标准写，不按写 README 的标准写。**

### 提示词式写法要点


| 原则               | 说明                                                                                                  |
| ---------------- | --------------------------------------------------------------------------------------------------- |
| **肯定式优先**        | 说做什么、用什么；不顺便列举未发生的错误路径                                                                              |
| **不说反面 priming** | ❌ `用 pnpm --filter <pkg>，不要根目录盲跑` → ✅ `跑包内任务：pnpm --filter <pkg> <script>`。不提「根目录盲跑」，Agent 就不会想到那条路 |
| **一条一动作**        | 一条一个可执行意图                                                                                           |
| **具体锚点**         | 命令、路径、文件名写全；避免「适当」「尽量」                                                                              |
| **只写限制和做法**      | 不写理由、不做人话 filler（「如你所知」「显然」）                                                                        |
| **无元叙事**         | 不写「本节说明…」                                                                                           |


### 何时才写「不要 X」（红线）

仅当 **X 是已验证的高频误行、且不说就会反复犯** 时。若错误路径本不存在于 Agent 行为中，**不提即可**。

## 4. 红线与绿线


|        | **红线**           | **绿线**                     |
| ------ | ---------------- | -------------------------- |
| **语气** | 不要 / 禁止 / 未经许可不… | 优先 / 用 X / 先…再… / 放这里      |
| **目的** | 避免代价大的错误         | 提高质量与效率                    |
| **例子** | 不要 commit `.env` | `pnpm --filter <pkg> test` |
| **例子** | 无用户要求不要 commit   | 改技能前加载 `skill-creator`     |
| **例子** | 不要改 `generated/` | 新页面放 `src/features/`       |
| **反例** | —                | 「写好代码」「多写测试」（通用废话）         |


绿线同样是**仓库事实 + 可执行做法**，只是导向推荐路径。

## 5. 应写什么 / 不应写什么

### 应写（高信号）


| 类别         | 说明                                               |
| ---------- | ------------------------------------------------ |
| **少许定向**   | 2–4 句：是什么仓、边界在哪；服务于干活定向，不讲故事                     |
| **关键节点目录** | 浅层、为**摆放文件**（像工程师分门别类），**不是**为检索；大项目深层目录树无效      |
| **命令**     | 非显而易见的、可复制的完整命令（带 flag）                          |
| **绿线**     | 本仓最高效、最正确的做法                                     |
| **红线**     | 真雷、高频误行                                          |
| **路由**     | 指向 Skill / docs / 嵌套 `AGENTS.md` 的指针（**引用，非复制**） |


### 不应写


| 类别                     | 原因                              |
| ---------------------- | ------------------------------- |
| 深层目录树 / 逐文件说明          | 检索无效；占满上下文                      |
| README / manifest 已有内容 | 冗余、易过期、研究证实有害                   |
| 通用最佳实践                 | 模型已知或属 user rules               |
| 完整工作流正文                | 属 Skill，最多一行路由                  |
| 长篇架构 / API             | 属 `docs/`，链接即可                  |
| 对比式噪声                  | 「用 A 不要 B」中的 B 可能 priming 非预期行为 |
| 删改记录 / 「此处已删除…」        | 不是流水账；删了 Agent 看不到，不必解释         |


### 判断一问

> 这条约定，换一个人、换一个 Agent 工具，是否仍应成立？

- 团队仓库：个人偏好（如回复语言）应分层；单人维护仓可务实处理
- 否 → 不进 `AGENTS.md`（或进 user rules）

## 6. 与其他载体的分工

各司其职，**同一事实只维护一处**。


| 载体                  | 放什么                                   |
| ------------------- | ------------------------------------- |
| **AGENTS.md**       | 跨工具、全仓（或子树）常驻：定向、命令、绿线/红线、路由          |
| **Agent Skills**    | 完整可触发工作流；须在 `AGENTS.md` **引用**否则不自动触发 |
| `**.cursor/rules**` | Cursor 专属、可按 glob 条件生效                |
| `**CLAUDE.md**`     | Claude Code 专属；可用 `@AGENTS.md` 导入共享内容 |
| **README / docs**   | 人类可读 + Agent 可自行检索的深度材料               |


**引用 ≠ 复制**：链过去，不要把 Skill/Rule 全文抄进 `AGENTS.md`。

工作流落点视仓库与 Agent 工具而定；`AGENTS.md` 负责让 Agent 知道**去哪加载**。

## 7. 嵌套与体积

### 根 vs 嵌套

- **根 `AGENTS.md`**：全仓库适用的约定
- **嵌套 `AGENTS.md`**：子树特有约定；**善于主动使用嵌套**
- 嵌套文件只写**增量**，不抄父文件
- 子级优先；与官方「nearest wins」一致

### 「极短」是纪律，不是行数 KPI

- 项目有大有小，不卡死字数
- 控制的是**写作规范**和**信号密度**
- 真能写到很长，说明早该拆分嵌套或迁出，而不是继续堆根文件

### 结构

- **无固定章节模板**；有什么写什么，没内容不凑章
- 常见健康形态（非 checklist）：定向 → 关键目录（可选）→ 命令 → 绿线/红线 → 路由
- 维护时**优先改原段落**，不轻易加新一级标题

## 8. 何时读 / 写 / 改 / 不写

### 读 / 遵循（不加载本技能）

- 进入陌生仓库、按约定干活时：依赖工具已注入的 `AGENTS.md`，**直接照做**即可
- 不需要为「遵守 AGENTS.md」而加载 `using-agentsmd`

### 写（必须加载本技能）

- 仓库尚无文件，且已有**稳定的、非显而易见的**约定需沉淀
- 用户明确要求
- `/init` 类：先搜集仓库信息 → **对话对齐**（有 brainstorming 等插件则用）→ 再落笔；不一键吐全文

### 改（必须加载本技能）

- 约定变更、指令过时、发现冗余/废话
- 仓库配置变更时，可**提醒**同步修改；实际改文件前**必须加载本技能**

### 不写 / 不改

- 无用户提示时不主动改；要改须**明确要求**或**先征得许可**
- 仓库没有 `AGENTS.md` 时不默认「缺了就得补」
- 内容可推断、属工作流/深度文档、非必要不追加

### 编写时

- 对照 `package.json`、CI、Makefile、真实目录等写入**真事实**；不猜、不抄别仓

### 执行时

- **直接照 `AGENTS.md` 做**；不先校验浪费 token；有问题再处理

## 9. 审计与维护纪律

改已有文件时，默认是**编辑器，不是扩写器**：

1. **先删**：重复、过时、overview、废话、通用说教
2. **再迁**：错放内容移到 Skill / Rule / docs，留路由
3. **最后才补**：真正缺且非可推断的条目
4. **删就是删**：不附「此处已删除…」；不当流水账
5. **专注眼前**：只描述当前怎么干活，不记录历史
6. 问题被代码/工具吸收后，**从文件删掉**对应条

## 10. 内容原则（仓库事实）

- 写**这个项目实际怎么做**，不是 Agent 理想中该怎么做
- 不把 user rules 或通用准则抄进来
- 语言跟**仓库维护者、现有文档习惯**一致；技能正文英文，不强制 `AGENTS.md` 英文
- 用户当前对话的**明确指令** > `AGENTS.md` > Agent 默认习惯

## 11. 技能自身设计要点

### 结构（方案 A，推荐）

```
skills/using-agentsmd/
├── SKILL.md
└── references/
    ├── good-examples.md
    └── bad-examples.md
```

### SKILL.md 建议章节（轻量）

1. What & Why — 角色、prompt 本质、效率价值
2. Scope — 放什么 / 不放什么 / 分层
3. Prompt discipline — 肯定式、无 priming、只写做法
4. Red / Green lines — 概念 + 极简对照
5. When — 何时**必须加载本技能**（仅创建/修改）/ 何时不写 / 许可
6. How — 判断流（归属 → 草稿 → 自检 → 执行）
7. Maintenance — 先删后补、嵌套拆分
8. Examples — 指针到 references

### description 要点（实现时）

- 触发范围：**创建、修改、审计、init 生成** `AGENTS.md` 时加载；**不要**写成「任何涉及 AGENTS.md 的任务都加载」
- 明确 **Hard gate**：动手改文件前必须先加载本技能
- 不写「日常遵循 AGENTS.md 时也需加载」——那是被动上下文的事

### 正反例要求

- 每个 **10–20 行**，对照鲜明
- 附标签：目录堆砌、Skill 复制、对比 priming、删了又解释、凑模板空章…
- 正例可参考本仓库 `AGENTS.md`（路由 + 边界 + 轻量）

### 刻意不进技能正文

- 运行时注入 / 是否再 `read` 文件
- 安全专题控制
- 行数 KPI / 冗长 checklist
- 重复调研数据堆砌
